# Requisitos - Marco de Captura y Reconocimiento OCR

## 1. Objetivo

建立一套可落地的「文件拍攝與 OCR 辨識框架」，降低使用者因拍攝模糊、傾斜、反光、裁切不完整而造成的辨識失敗，提升整體辨識成功率與操作體驗。

本文件可作為以下用途：

- 產品需求文件
- 前端拍攝頁面規格
- 影像前處理與 OCR 串接依據
- 驗收與測試基準

## 2. 適用範圍

本框架適用於以下場景：

- 文件 OCR
- 單據／發票辨識
- 標籤／貼紙文字擷取
- 表單內容擷取
- 證件或卡片拍攝

第一階段以「文件與文字辨識」為主要目標，不包含複雜物件分類與多目標場景分析。

## 3. 使用者目標

使用者希望：

- 把文件放到拍攝框內
- 快速拍到可辨識的照片
- 系統自動判斷畫質是否合格
- 拍完後直接得到 OCR 結果
- 如果失敗，能清楚知道要怎麼重拍

## 4. 主要流程

1. 使用者開啟拍攝頁面
2. 畫面顯示中央辨識框與拍攝提示
3. 系統即時偵測文件邊界與品質
4. 若條件達標，啟用拍攝或自動拍攝
5. 拍攝後執行影像前處理
6. 將處理後影像送至 OCR 引擎
7. 顯示辨識結果
8. 若辨識品質不足，提示重拍

## 5. 拍攝頁面 UI 規格

### 5.1 版面結構

- 頁面上方顯示標題：`拍攝文件`
- 中央顯示主辨識框
- 框外顯示半透明遮罩
- 底部顯示拍攝提示與操作按鈕

### 5.2 辨識框規格

- 框寬建議為螢幕寬度的 `80% ~ 85%`
- 文件模式框比例建議為 `4:3` 或近似 `A4`
- 卡片模式可切換為 `1.58:1`
- 四角需有角標，用於提示對齊
- 框內不顯示多餘裝飾，避免干擾預覽

### 5.3 畫面提示文案

預設提示文案：

```text
請將文件完整放入框內
保持平整、避免反光
對準後即可拍攝
```

若支援自動拍攝，可改為：

```text
請將文件完整放入框內
保持平整、避免反光
對準後將自動拍攝
```

### 5.4 操作元件

- 返回按鈕
- 閃光燈開關
- 相簿匯入按鈕
- 拍攝按鈕
- 模式切換按鈕（文件／卡片，若有需要）
- 狀態提示區塊

## 6. 即時拍攝檢查條件

系統在拍攝前需持續檢查以下項目：

- 是否偵測到主要文件邊界
- 文件是否完整落在辨識框內
- 圖像是否清晰
- 亮度是否足夠
- 是否有明顯反光
- 文件傾斜角度是否過大
- 文件大小是否足夠支援 OCR

## 7. 品質判定規則

以下為第一版建議門檻，可於實作後依資料調整：

### 7.1 文件位置與大小

- 文件需落於主框內
- 文件面積需占框內區域 `70% ~ 95%`
- 文件四個角至少需穩定偵測到 `3 ~ 4` 個

### 7.2 傾斜角度

- 文件傾斜角度需小於 `10°`
- 若超過 `10°`，提示使用者重新對正

### 7.3 清晰度

- 需通過模糊度門檻
- 建議以 Laplacian variance 或等效方法檢測
- 若低於門檻，提示重新對焦

### 7.4 亮度

- 圖像平均亮度需落在合理區間
- 過暗時提示移至亮處
- 過亮時提示調整光源或拍攝角度

### 7.5 反光

- 若文件表面高亮區塊過大或覆蓋文字區域，應判定為不可辨識
- 需提示使用者調整角度，避免燈光直接反射

## 8. 拍攝控制邏輯

### 8.1 手動拍攝模式

- 當品質不合格時，仍可保留拍攝按鈕，但需提示可能辨識失敗
- 當品質達標時，狀態顯示為 `可拍攝`

### 8.2 自動拍攝模式

- 僅在下列條件持續穩定 `0.5 ~ 1.0` 秒後自動拍攝：
- 文件完整入框
- 角度合格
- 清晰度合格
- 亮度合格
- 無明顯反光

## 9. 拍攝後影像前處理

拍攝完成後，在送 OCR 前需執行以下步驟：

1. 偵測文件輪廓
2. 自動裁切文件區域
3. 透視校正
4. 灰階化
5. 對比增強
6. 去噪
7. 必要時進行二值化
8. 文字區域銳化

目標是提高 OCR 對文字邊界、低對比文字與斜拍影像的辨識能力。

## 10. OCR 辨識流程

建議處理流程：

1. 相機拍攝原圖
2. 執行前處理
3. 將處理後影像送至 OCR 模組
4. 回傳文字結果與信心分數
5. 若信心分數低於門檻，標示為低可信度
6. 視需要提示使用者重拍或手動修正

## 11. 錯誤提示文案

系統應依失敗原因給出明確提示，不使用模糊訊息。

建議文案如下：

- 畫面太模糊，請重新對焦後拍攝
- 光線不足，請移到更亮的地方
- 反光過強，請調整拍攝角度
- 文件未完整入鏡，請將四個角放入框內
- 手機太斜，請對正文件後再拍
- 文件太遠，請靠近一些再拍
- 暫時無法辨識內容，請重新拍攝一張更清楚的照片

## 12. 驗收標準

第一版建議驗收條件：

- 使用者可在拍攝頁面清楚看到辨識框與拍攝提示
- 系統可即時提示模糊、反光、過暗、傾斜、未完整入鏡等問題
- 拍攝後可自動裁切文件區域
- 拍攝後可完成透視校正與基礎影像增強
- OCR 可成功輸出文字結果
- 錯誤狀況下可顯示對應提示文案

## 13. 建議資料結構

前端或後端可統一使用以下欄位描述一次拍攝結果：

```json
{
  "documentDetected": true,
  "insideFrame": true,
  "blurScore": 124.5,
  "brightnessScore": 0.68,
  "glareDetected": false,
  "tiltAngle": 4.2,
  "captureReady": true,
  "ocrConfidence": 0.91,
  "message": "可拍攝"
}
```

## 14. 建議狀態機

可將拍攝流程拆成以下狀態：

- `idle`
- `detecting`
- `misaligned`
- `blurry`
- `too_dark`
- `glare`
- `ready`
- `capturing`
- `processing`
- `ocr_success`
- `ocr_failed`

這樣可降低前端與辨識服務之間的邏輯混亂。

## 15. 後續擴充方向

第二階段可擴充以下能力：

- 多語系 OCR
- 表格欄位結構化擷取
- 發票／單據欄位解析
- 證件正反面引導
- 批次拍攝
- 自動連拍挑最佳張
- 拍攝品質分數儀表

## 16. 實作建議

如果要快速落地，建議分三層實作：

### 16.1 前端層

- 相機預覽
- 辨識框 UI
- 即時狀態提示
- 手動或自動拍攝

### 16.2 影像處理層

- 邊界偵測
- 透視校正
- 去噪與增強
- 品質評估

### 16.3 OCR 服務層

- 文字辨識
- 信心分數回傳
- 結果整理與欄位解析

## 17. 一句話需求摘要

系統需提供一個文件拍攝辨識框架，能在拍攝前即時檢查文件是否清晰、完整、對正且無反光，拍攝後自動進行裁切與校正，再送 OCR 辨識，並於失敗時清楚引導使用者重拍。

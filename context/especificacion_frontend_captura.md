# Especificacion Frontend - Flujo de Captura OCR

## 1. Objetivo

本文件定義文件拍攝頁的前端互動與狀態流，讓使用者能在最短時間內拍出可辨識的照片，並在品質不足時立即收到可執行的提示。

## 2. 頁面目標

- 提供清楚的拍攝框與引導文案
- 即時顯示拍攝品質狀態
- 在條件達標時允許手動或自動拍攝
- 拍攝後呈現處理中與辨識結果
- 失敗時能快速重拍

## 3. 核心畫面區塊

### 3.1 頂部區塊

- 返回按鈕
- 畫面標題：`拍攝文件`
- 模式標籤：`文件 OCR`

### 3.2 中央預覽區

- 相機預覽畫面
- 中央辨識框
- 四角對齊角標
- 框外遮罩
- 即時偵測狀態浮層

### 3.3 底部操作區

- 狀態提示文字
- 品質指標列
- 閃光燈按鈕
- 相簿匯入按鈕
- 拍攝按鈕
- 自動拍攝開關

## 4. 主辨識框規格

- 寬度：畫面寬度 `82%`
- 高度：依文件比例顯示，預設 `4:3`
- 圓角：`20px`
- 框線粗細：`2px`
- 預設顏色：白色半透明
- 當狀態 `ready` 時改為綠色
- 當狀態異常時改為橘色或紅色

## 5. 品質指標

畫面底部顯示四項簡化品質指標：

- 對齊
- 清晰
- 光線
- 反光

每項指標只有三種狀態：

- `ok`
- `warning`
- `error`

## 6. 前端狀態機

建議以前端 state machine 管理拍攝流程：

```text
idle
detecting
misaligned
blurry
too_dark
glare
ready
capturing
processing
success
failed
```

## 7. 狀態對應 UI

### 7.1 `idle`

- 顯示初始提示
- 拍攝按鈕可見但不強調

### 7.2 `detecting`

- 顯示 `正在偵測文件位置`
- 框線維持中性顏色

### 7.3 `misaligned`

- 顯示 `請將文件完整放入框內`
- 框線改為橘色

### 7.4 `blurry`

- 顯示 `畫面太模糊，請重新對焦`
- 清晰指標轉為 `error`

### 7.5 `too_dark`

- 顯示 `光線不足，請移到更亮的地方`
- 光線指標轉為 `warning` 或 `error`

### 7.6 `glare`

- 顯示 `反光過強，請調整拍攝角度`
- 反光指標轉為 `error`

### 7.7 `ready`

- 顯示 `已對準，可拍攝`
- 框線改為綠色
- 若啟用自動拍攝，顯示倒數或穩定中狀態

### 7.8 `capturing`

- 暫停互動
- 顯示短暫拍攝動畫

### 7.9 `processing`

- 顯示 `正在校正與辨識`
- 禁止重複提交

### 7.10 `success`

- 顯示結果摘要與操作按鈕
- 提供 `重新拍攝` 與 `確認送出`

### 7.11 `failed`

- 顯示失敗原因
- 提供 `重新拍攝`

## 8. 文案規則

前端所有提示文案需符合以下原則：

- 短句
- 明確指出問題
- 直接告訴使用者下一步
- 不使用技術術語

範例：

- `請將文件四個角放入框內`
- `請靠近一些再拍`
- `請先對焦再拍攝`
- `文字被反光遮住，請換角度`

## 9. 使用者流程

1. 進入拍攝頁
2. 看見中央辨識框與引導文案
3. 將文件移至框內
4. 根據即時指標調整角度、距離與光線
5. 系統判定 `ready`
6. 使用者點擊拍攝，或系統自動拍攝
7. 前端顯示 `processing`
8. 顯示 OCR 結果或重拍提示

## 10. 回傳資料需求

前端應能接收以下拍攝品質資料：

```json
{
  "captureState": "ready",
  "documentDetected": true,
  "insideFrame": true,
  "blurScore": 138.2,
  "brightnessScore": 0.74,
  "glareDetected": false,
  "tiltAngle": 3.8,
  "message": "已對準，可拍攝"
}
```

## 11. 成功結果頁最低需求

- 顯示已校正影像縮圖
- 顯示 OCR 抽出的文字片段
- 顯示信心分數或品質標示
- 提供 `重新拍攝`
- 提供 `確認使用`

## 12. 響應式規則

- 手機為主
- 小尺寸裝置優先保留中央預覽區
- 底部操作區不得擋住主辨識框
- 橫向模式可作為第二階段支援

## 13. 無障礙建議

- 主要按鈕需有足夠點擊面積
- 狀態提示不只靠顏色，也要有文字
- 按鈕與提示需支援螢幕閱讀器標籤

## 14. 驗收重點

- 使用者能在 3 秒內理解如何對準文件
- 品質不佳時能收到對應提示
- 狀態切換不混亂、不跳閃
- `ready` 與 `processing` 的差異清楚
- 成功與失敗後都有明確下一步

# 住院模式 v2 — 設計檢視報告（Design Critique）

> 版本：2026-06-05 ｜ 範圍：現有住院模式（v1 六大功能 + Phase 1 M07）整體 UX ＋ Phase 2（M01 ePRO / M04 問醫師問題）設計方向
> 依據：實際程式碼 `frontend/js/app.js`（`renderInpatientHome` 等）、`backend/main.py` include_router 清單、`MD_Piece_住院模式PWA設計計畫.md`、`docs/product-principles.md` 7 條設計憲法
> 方法：現況 IA 盤點 → 問題分級（P0 阻斷 / P1 重要 / P2 打磨）→ 對照設計憲法與計畫書 NFR → 重設計方向

---

## 0. 一句話結論

> 現有住院模式**功能很全、但首頁過載、缺少「核心 ePRO 迴圈」與「查房前問醫師」這兩個住院情境最高價值的流程**；症狀輸入有三套並存的心智模型衝突。Phase 2 不該只是「再加兩個模組」，而是**藉 M01/M04 把住院模式收斂成一條清楚的每日主迴圈**。

---

## 1. 現況資訊架構盤點

`renderInpatientHome(ctx)` 由上而下堆疊 **9 個區塊**：

| # | 區塊 | 函式 | 性質 |
|---|---|---|---|
| 1 | 現在狀態 NowCard | `renderInpatientNowCard` | 狀態 hero |
| 2 | 下一步 NextStep | `renderInpatientNextStep` | 任務倒數 |
| 3 | 住院工具 Hub（5 卡） | `renderInpatientHub` | 導覽 |
| 4 | 一鍵回報 SOS（痛/喘/噁心/求助） | `renderInpatientSOS` | 症狀輸入 ① |
| 5 | 今日治療 Timeline | `renderInpatientTimeline` | 排程 |
| 6 | 今日進度 Ring + 查房摘要 | `renderInpatientDayProgress` / `RoundsCard` | 進度 / 紀錄 |
| 7 | 病情趨勢（痛/累/心情，quicklog） | `renderInpatientTrendChips` | 症狀輸入 ② |
| 8 | 出院進度 stepper | `renderInpatientDischargeStepper` | 出院 |
| 9 | 住院衛教 | `renderInpatientEducation` | 教育 |

Hub 內另有 5 個工具：交接報告、用藥核對、**床邊紀錄（痛/睡/吃 + 想問醫師）← 症狀輸入 ③**、住院衛教、出院準備。

設計計畫書 §2 對「Today」的定義只有 **4 件事**：今日提醒、症狀趨勢迷你圖、待辦（1 ePRO + 3 問題）、1 段衛教影音。**現況比計畫書多了一倍以上的首頁內容。**

## 2. 發現（分級）

### 🔴 P0 — 阻斷級（不解，Phase 2 KPI 無法達成）

**P0-1 首頁過載，違反「精簡 Today」與設計憲法 6（長者／家屬模式）**
9 個區塊垂直堆疊，低 eHL／長者使用者一屏看不完、不知道「現在到底要我做什麼」。計畫書 NFR 訂 SUS ≥ 70（目標 80），過載是 SUS 的頭號殺手。
→ Today 應收斂成「**一個主行動 + 三個輔助瞥一眼**」，其餘進分頁。

**P0-2 缺少核心 ePRO 迴圈（M01）**
計畫書把 ePRO 列為**核心模組**（§2「自陳(ePRO)— 核心模組」），定義為「每日 1 次推播 → 5 題量表 + 1 自由描述（可語音/拍照）」。現況只有：
- SOS：痛/喘/噁心/求助（4 顆即時鈕，無量表、無每日結構）
- quicklog：pain/fatigue/mood（1–5，點趨勢 chip 才觸發，非每日推播）

兩者都**不是**結構化的每日 ePRO。計畫書 MVP 驗收：「Day 3 至少 1 次 ePRO 完成率 ≥ 60%」——**沒有 ePRO 模組，這條 KPI 無從量測**。

**P0-3 M04「問醫師的問題」沒有第一級入口**
查房前準備是住院最高價值、最焦慮的時刻（計畫書 Flow B「查房前 5 分鐘」）。現況「想問醫師」只存在於 Hub → bedside 工具的**一句描述文字**裡，沒有：候選題池、從 ePRO 自動建議、勾選優先序、查房當天一鍵展開朗讀卡。→ Phase 2 的 M04 等於從零做。

### 🟠 P1 — 重要（傷害一致性與信任）

**P1-1 症狀輸入三套並存，違反鐵則 7（衝突要攤開，別混合）**
SOS（痛/喘/噁心）、quicklog（痛/累/心情）、bedside（痛/睡/吃）——三套不同維度、不同入口、資料各自落地。患者要學三種心智模型，資料無法彙整成一條趨勢，也餵不進 M04 的「自動建議問題」。
→ **收斂為單一 ePRO 模型**：ePRO（每日結構化）為主，SOS 退化為 ePRO 的「即時不適快捷」，quicklog 併入 ePRO 題項。

**P1-2 SOS「已通知護理站」是 UI 假承諾，與計畫書 §1.2「不取代護理鈴」抵觸**
程式碼註解自承「後端串接前先當 UI 假承諾，避免病人焦慮等不到回應」。但讓病人**誤以為護理站已收到**，在臨床情境是安全與信任風險（病人可能因此不按實體叫人鈴）。違反設計憲法 2（可信任、可解釋）。
→ 文案改為誠實版：「已記到你的紀錄裡，查房時可以給醫師看。**緊急請按床邊紅色叫人鈴**」，並把實體叫人鈴指引放在 SOS 旁。

**P1-3 記錄 vs 準備的概念混淆**
「查房摘要 RoundsCard」（記醫師說了什麼，≈ M08 我見了誰）和 M04（準備我要問的）被放在一起，但一個是**事後紀錄**、一個是**事前準備**，方向相反。應在 IA 上明確分流。

### 🟡 P2 — 打磨

- **P2-1 模組重複**：衛教（首頁 §9 + Hub）、出院（首頁 §8 stepper + Hub dischargePlan）各出現兩次入口。
- **P2-2 裝飾性編號**：section 的 `01–07`（`ip-num`）對患者無資訊量，徒增視覺噪音；保留給內部 code comment 即可。
- **P2-3 無障礙抽查**：`ip-trend-explain` 用 `event.stopPropagation()` 擋冒泡，鍵盤使用者無法觸發說明；裝飾數字需確認 `aria-hidden`。

## 3. 對照設計憲法（7 條）落差

| 憲法 | 現況 | 落差 |
|---|---|---|
| 1 PWA 原生體驗 | ✅ 已是 PWA | — |
| 2 可信任、可解釋 AI（附「為什麼」） | ⚠️ | SOS 假承諾；趨勢 chip 缺「為什麼變化」說明 |

## 4. 重設計方向（為 Phase 2 鋪路）

### 4.1 新資訊架構 — 一條每日主迴圈

```
住院模式（底部 tabbar 隨 care mode）
├─ 今日 Today（精簡：一個主行動 + 三個瞥一眼）
│   ├─ 現在狀態（住院第 N 天 · 一句安心話）
│   ├─ ★主行動卡：今天的 ePRO（未填→「填今天」；已填→趨勢摘要）
│   ├─ 瞥一眼：下一個排程 / 趨勢迷你 / 1 衛教
│   └─ 即時不適快捷（痛/喘/噁心，誠實文案 + 叫人鈴指引）
├─ 自陳 ePRO（M01）★Phase 2
│   └─ 每日 5 題量表 + 1 自由描述（語音/拍照）→ 提交即生成問醫師候選
├─ 問醫師 Questions（M04）★Phase 2
│   ├─ 從 ePRO 自動建議的候選題
│   ├─ 自己加題 + 勾選今天想問的 3 題
│   └─ 查房前一鍵展開「朗讀卡」
├─ 我的狀況（用藥核對 / 交接報告 / 今日治療 / 查房紀錄）
└─ 更多（出院準備 / 衛教 / 設定字級·語言）
```

### 4.2 兩條 Phase 2 主流程

**Flow M01 — 每日 ePRO（≤6 步、可語音）**
```
推播「今天覺得怎麼樣？1 分鐘」→ 5 題滑桿/表情（痛 NRS 0–10、累、噁心、睡、心情）
→ 1 題自由描述（可按麥克風語音、可拍照）→ 送出 → 「謝謝，幫你記下了」
→ 自動把異常項（如痛 ≥ 7）轉成「要問醫師」候選
```

**Flow M04 — 查房前 5 分鐘（計畫書 Flow B）**
```
推播「醫師快來了，要準備嗎？」→ 顯示 24h ePRO 摘要
→ 候選問題清單（ePRO 自動建議 + 自己加）→ 勾 3 題
→ 「朗讀卡」大字模式：醫師進來時翻開唸「我有 3 件想問…」
```

### 4.3 收斂原則（手術式，鐵則 3）

1. **症狀三套 → 一套**：ePRO 為單一真相；SOS 改為 ePRO 的即時快捷、quicklog 題項併入 ePRO。**保留 SOS 的「一鍵」精神**（住院常無力打字），但資料同源。
2. **首頁 9 區 → Today 4 件事**：其餘移到 tab，不刪功能、只搬位置。
3. **誠實文案**：移除「已通知護理站」假承諾，補實體叫人鈴指引。
4. **編號裝飾移除**：`ip-num` 退回 code comment。

### 4.4 不做（避免過度工程，鐵則 2）

- 不重寫 v1 六大功能後端（`inpatient.py` / `admissions.py` 保留不動）。
- 不在這階段做 OCR（M02/M03，屬計畫書 Phase 2 後段 / 本實作 Phase 3）。
- 不做家屬代理（M09，Phase 5）。

## 5. 驗收對齊（計畫書 §10 NFR）

| 指標 | 目標 | 重設計如何達成 |
|---|---|---|
| SUS | ≥ 70（目標 80） | 首頁收斂、單一症狀模型降低學習成本 |
| ePRO Day3 完成率 | ≥ 60% | M01 每日推播 + Today 主行動卡常駐提醒 |
| 查房問題數 | ≥ 2 題/人/查房前 | M04 自動建議 + 一鍵展開降低門檻 |
| WCAG 2.1 AA | 全頁通過 | 沿用 M07 a11y 模式（radiogroup/focus/Esc），修 P2-3 |

# Storm / Co-Storm 在醫療衛教中的角色

> 記錄日期：2026-03-26
> 關聯專案：MD.Piece — PWA 醫療輔助平台

---

## 一、問題背景：醫療衛教文章的困境

傳統生成醫療衛教文章面臨三大挑戰：

1. **需要大量、精準、有引用來源的長文**
   醫療內容不能只是「大概說得通」，必須有文獻佐證、來源可查驗，否則無法取得醫療專業人員與患者的信任。

2. **單一 LLM 容易幻覺（Hallucination）**
   單次呼叫語言模型時，模型可能「捏造」不存在的研究、錯誤的藥物劑量或過時的治療指引，在醫療場景中風險極高。

3. **缺乏結構與組織**
   長篇衛教文章需要清晰的大綱層次（症狀 → 診斷 → 治療 → 照護 → 預防），單次生成很難維持一致的邏輯結構。

---

## 二、Storm 如何解決這些問題

### Storm 的核心機制

**Storm（Synthesis of Topic Outlines through Retrieval and Multi-perspective Question Asking）** 是斯坦福 NLP 實驗室開發的框架，模仿維基百科的編輯流程：

```
主題輸入
   ↓
多輪網路搜尋（模擬不同視角的提問）
   ↓
自動蒐集多方來源與引用
   ↓
組織大綱（章節層次化）
   ↓
產出有結構的長篇知識文章（含引用標注）
```

### 具體優勢

| 問題 | Storm 的解法 |
|------|-------------|
| 需要引用來源 | 每段論述自動標注來源 URL / DOI |
| 幻覺風險高 | 多輪搜尋 + 引用驗證，降低無根據陳述 |
| 缺乏結構 | 先產大綱再填內容，層次清晰 |
| 單一視角 | 模擬多個「提問者角色」從不同角度調研 |

## 三、Co-Storm：適合醫療場景的協作版本

**Co-Storm** 是 Storm 的協作延伸版本，核心差異在於：

- **支援人類專家介入**：醫師、藥師或衛教師可以在生成過程中插入意見、修正方向
- **共同策展（Co-curation）**：AI 負責調研與初稿，人類專家負責審核與最終把關
- **適合有審核需求的醫療場景**：符合「AI 輔助、人類負責」的醫療合規原則

```
AI 調研層（Storm）
     ↓
草稿產出
     ↓
醫師 / 衛教師審核介入 ←── Co-Storm 的關鍵環節
     ↓
最終核准發布
```

## 四、在 MD.Piece 整體架構中的位置

MD.Piece 採用四層架構，Storm / Co-Storm 位於最頂層的「自動研究層」：

```
┌─────────────────────────────────────────────────────┐
│  自動研究層                                          │
│  • Storm / Co-Storm  → 生產衛教知識長文              │
│  • gpt-researcher    → 蒐集文獻與研究報告            │
└──────────────────────┬──────────────────────────────┘
                       │ 產出結構化知識文章 + 引用
                       ↓
┌─────────────────────────────────────────────────────┐
│  知識庫層                                            │
│  • Dify / Flowise    → RAG，讓 AI 問答有根據         │
│    （將衛教文章向量化，作為 AI 問答的知識來源）       │
└──────────────────────┬──────────────────────────────┘
                       │ 精準的語意檢索結果
                       ↓
┌─────────────────────────────────────────────────────┐
│  使用者互動層                                        │
│  • Rasa / Botpress   → 症狀問答 Chatbot              │
│    （根據知識庫回答病患問題、引導就診）               │
└──────────────────────┬──────────────────────────────┘
                       │ 對話結果 / 視訊診療需求
                       ↓
┌─────────────────────────────────────────────────────┐
│  介面層                                              │
│  • chat-ui-kit-react → 對話 UI                       │
│  • Jitsi             → 視訊看診                      │
└─────────────────────────────────────────────────────┘
```

### 各層職責說明

| 層次 | 工具 | 在 MD.Piece 的角色 |
|------|------|-------------------|
| 自動研究層 | Storm, gpt-researcher | 自動產生有根據的衛教長文 |
| 知識庫層 | Dify, Flowise | 將文章轉為可查詢的向量知識庫（RAG） |
| 互動層 | Rasa, Botpress | 症狀收集、衛教問答、就診引導 |
| 介面層 | chat-ui-kit-react, Jitsi | 病患與醫師的實際操作介面 |

---

## 五、為什麼這個架構組合重要

1. **可信度鏈**：從「有引用的文章（Storm）」→「向量化知識庫（RAG）」→「問答機器人」，確保每一層的答案都有原始來源可追溯。

2. **人機協作符合醫療合規**：Co-Storm 的醫師審核節點，讓最終發布的衛教內容經過專業把關，降低法規風險。

3. **分層可替換**：各層工具可獨立升級（例如將 Dify 換成其他 RAG 平台），不影響整體架構。

4. **減少重複工作**：衛教文章一次生成，透過 RAG 多次複用，醫師不需要每次對話都重新解釋相同的疾病知識。
